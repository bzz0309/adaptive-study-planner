# 多考试学习陪伴与成长应援系统产品文档

版本：v0.1  
更新时间：2026-06-23  
项目仓库：https://github.com/bzz0309/adaptive-study-planner  
线上奖励系统预览：https://reward-companion-system.vercel.app

## 1. 产品概述

本产品是一个面向多种考试和长期学习目标的学习陪伴系统。

它不是单纯的学习打卡工具，也不是普通课程表生成器，而是一个将「考试 / 学习目标配置」「自适应学习规划」「每日陪伴」「阶段奖励」结合在一起的成长应援系统。

用户通过填写考试或学习项目的信息生成学习计划；在每天完成学习、打卡、解决错题或达成连续学习节点后，系统以温柔、陪伴式的方式给出反馈，并通过紫色演唱会应援风格的奖励场景，让用户感受到「自己的坚持被看见」。

产品核心体验可以概括为：

> 用户不是在被提醒学习，而是在每一次坚持后，进入一场只为她亮起的紫色应援。

## 2. 产品定位

### 2.1 产品类型

多考试学习规划 + 学习陪伴 + 情绪奖励系统。

### 2.2 产品关键词

- 多考试备考
- 学习计划
- 每日打卡
- 明日重点
- 错题解决
- 成长记录
- 情绪奖励
- 紫色应援
- 泰蛋
- K-pop 演唱会氛围
- 温柔陪伴

### 2.3 产品调性

产品语气应当柔和、肯定、有陪伴感。

不应像系统在下达任务，也不应像普通工具在机械反馈。用户完成学习后，反馈应该更像有人轻轻说：「今天也有好好往前走一点」。

## 3. 目标用户

### 3.1 核心用户

正在准备某一类考试、语言考试、资格考试，或长期学习项目的学习者。

当前主项目已支持：

- TOPIK
- IELTS
- 其他自定义考试或学习项目

TOPIK 是当前讨论最深入、实现细节最多的重点场景，但不是产品唯一边界。

### 3.2 用户特征

- 有明确考试目标，但容易在长期备考中疲惫
- 需要计划感，也需要情绪支持
- 喜欢阶段性记录和仪式感
- 对冷冰冰的任务提醒接受度较低
- 对泰妍、紫色应援、泰蛋、应援棒、演唱会灯海等元素有情感连接

### 3.3 核心需求

- 明确知道自己接下来怎么学
- 每天能轻量记录学习进展
- 打卡后自动获得明日重点
- 在坚持学习时被鼓励，而不是被催促
- 阶段性成果能被纪念和收藏

## 4. 产品目标

### 4.1 功能目标

- 帮助用户根据考试类型、目标、考试时间和学习资料生成学习计划
- 支持每日学习打卡
- 根据打卡结果自动生成明日学习重点
- 在关键学习节点触发奖励系统
- 支持后续扩展多种奖励类型

### 4.2 情绪目标

- 让用户觉得自己的学习不是孤单推进的
- 让每一次完成都获得柔和、明确的肯定
- 将长期学习拆解为可被纪念的成长节点
- 用应援演唱会的方式强化「被陪伴」和「被看见」的感受

## 6. 主项目功能需求

### 6.1 学习信息填写

用户需要填写或选择以下信息：

- 考试名称：TOPIK / IELTS / 其他
- 考试类别或学习项目类型
- 目标等级、目标分数或目标描述
- 考试日期、考试届次或学习截止时间
- 当前学习基础或备考状态
- 学习强度：轻量、中等、自定义等
- 可用学习时间
- 每周学习日
- 薄弱内容
- 学习资料或资料链接

不同考试对应不同字段：

- TOPIK：区分 TOPIK I / TOPIK II，并支持目标等级
- IELTS：区分 Academic / General Training，并围绕听、说、读、写四项规划
- 其他：支持用户自定义考试名称、学习项目名称和目标

### 6.1.1 学习强度与时间配置

学习计划不能只根据考试日期生成，还需要考虑用户真实可用时间。

当前主项目已包含：

- 学习强度选择
- 每天最少 / 最多学习时长
- 可学习时间段
- 每周学习日
- 第一轮计划用时

计划生成时应根据这些信息控制任务密度，避免生成看起来完整但实际不可执行的计划。

### 6.2 关键信息核对

系统需要在生成计划前引导用户核对已填写或选择的所有关键信息。

文案原则：

不使用过窄表达，例如：

```text
请核对：TOPIK I/II、目标等级、考试日期或届次是否正确。
```

更合适的表达方向：

```text
请核对你刚才填写和选择的关键信息。
```

或：

```text
生成计划前，先一起确认这些信息有没有填对。
```

### 6.3 学习课程表生成

系统基于用户的考试目标、备考时间、薄弱内容和资料来源生成学习课程表。

课程表需要体现：

- 阶段目标
- 每日任务
- 复习安排
- 错题回看
- 考前冲刺节奏

对不同考试，计划内容应按考试结构自动调整。

例如：

- TOPIK I：听力、阅读、词汇语法基础
- TOPIK II：听力、写作、阅读、词汇语法基础
- IELTS：Listening、Reading、Writing、Speaking
- 其他考试 / 项目：根据用户填写内容、上传资料和 AI 搜索结果生成模块化计划

### 6.4 每日打卡

用户完成当天学习后可以打卡。

打卡不是为了制造压力，而是用于记录：

- 今天完成了什么
- 哪些内容卡住了
- 哪些错题解决了
- 明天应该优先看什么

### 6.5 明日重点生成

系统根据当天打卡结果生成明日重点。

用户曾提出的任务流方向：

```text
每天根据打卡自动生成明日重点
```

该能力应作为学习陪伴系统的重要闭环。

### 6.6 学习提醒

默认提醒时间当前定为：

```text
21:30
```

原因：

- 晚上 22:30 对用户可能偏晚
- 21:30 更适合作为轻量提醒
- 提醒语气应柔和，不应制造负担

### 6.7 错题集

错题集是主项目的核心学习闭环之一。

它不只是存放做错的题，而是帮助用户完成：

```text
错题进入系统 → 归因整理 → 到期复习 → 变式练习 → 判断是否掌握 → 影响明日重点 / 触发奖励
```

#### 6.7.1 错题来源

错题可以来自：

- 每日学习任务中的练习结果
- 模拟题或限时训练
- 用户手动导入的历史错题
- 上传的错题照片或 PDF
- 示例错题体验流程

当前原型中已包含「导入旧错题」入口，支持选择错题照片或 PDF。

#### 6.7.2 历史错题导入

历史错题导入用于解决用户已经有一批旧错题，但不想手动重新录入的问题。

当前规划 / 原型能力：

- 支持 JPG、PNG、WebP 照片
- 支持 PDF
- 一次最多导入 3 张
- 上传前自动压缩
- 识别不清楚时要求用户确认
- 识别后展示可导入题目列表
- 用户确认后导入错题集
- 导入后自动安排诊断题

当前原型文案方向：

```text
识别到 3 道错题
2道可直接导入，1道需要确认
确认导入错题集
```

正式版可接入图片识别与题目分析能力。

#### 6.7.3 错题信息结构

每道错题至少应包含：

- 错题标题
- 所属考试 / 项目
- 所属模块，例如听力、阅读、写作、口语、词汇、语法等
- 错题重点
- 错误原因
- 题目来源
- 是否到期复习
- 当前掌握状态
- 复习记录
- 变式练习记录

#### 6.7.4 错题复习队列

错题进入错题集后，不应只停留在列表里。

系统需要根据复习节奏安排：

- 当日复盘
- 次日重做
- 3 天后变式练习
- 7 天后延迟检验
- 14 天后再次确认

当前课程表中已经包含「错题复盘」类型任务，例如：

```text
否定表达错题复盘
本周错题二次检验
到期错题复盘
延迟变式检验
```

#### 6.7.5 变式练习

错题解决不等于看懂解析。

用户需要完成同类变式题，证明自己真的掌握了判断路径。

当前原型包含：

- AI 生成练习题
- 到期复习变式题
- 错题诊断练习
- 练习完成后自动记录正确率
- 错题结果写入学习打卡或错题集

示例流程：

```text
重做错题
说出判断依据
完成 5 道同类变式题
根据正确率判断是否继续复习
```

#### 6.7.6 掌握标准

错题标记为「已掌握」不能只依赖用户主观点击。

当前掌握标准方向：

```text
连续两次正确完成延迟变式题，并能讲明思路。
```

当错题达到掌握标准后：

- 从高频复习队列中移出
- 进入已掌握知识点统计
- 在之后再次答错时，自动恢复到错题复习队列

当前原型中已有「已掌握知识点」入口和「掌握记录」页面。

#### 6.7.7 明日重点联动

错题集会影响明日重点。

如果用户当天有未解决错题或练习错误，明日重点应优先安排：

- 重做对应错题
- 复述判断依据
- 完成同类变式题
- 回到对应知识点补弱

示例：

```text
明天先重做「否定表达漏听」错题：说出判断依据，再完成5道同类变式题。
```

#### 6.7.8 错题奖励触发

错题解决也应该触发奖励。

它和连续打卡奖励不同，代表的是「一个具体卡点被拿下了」。

当前已讨论过的奖励方向：

```text
高音通过
```

当前原型中的奖励文案方向：

```text
这道错题不是被跳过了，是被你慢慢拿下了。
```

后续可设计为：

- 首次解决错题
- 连续解决 3 道错题
- 一个知识点从错题队列移出
- 某类薄弱项首次达成掌握标准

对应奖励形态可以是：

- 高音通过
- 一枚小徽章
- 一张「我会了」小卡
- 一束短暂亮起的紫色追光

### 6.8 学习资料与 AI 核验

用户可以提供自己的学习资料，也可以让系统在资料不足时补充搜索。

#### 6.8.1 资料输入方式

当前主项目支持：

- 资料链接输入
- 本地资料上传
- 图片、PDF、Word、PPT、表格、文本等文件类型
- 一次最多上传 10 个学习资料文件
- 文件列表展示与删除

本地资料刷新后需要用户重新选择，不应默认长期保存原文件。

#### 6.8.2 AI 搜索补充

当用户资料不足时，可以开启：

```text
缺少资料时让 AI 搜索补充
```

搜索原则：

- 优先考试官网
- 优先权威资料
- 优先题型、考试范围、评分标准、官方样题
- 搜索结果需要展示来源
- 用户确认后才生成计划

#### 6.8.3 资料确认

生成计划前需要进入资料确认步骤。

确认内容包括：

- 考试 / 项目
- 目标
- 学习范围
- 时间安排
- 资料来源
- AI 搜索结果

文案方向：

```text
请最后核对一次关键信息。确认后，AI 会按这些资料生成课程表。
```

如果 AI 搜索仍在进行，应禁用生成按钮，并提示等待资料核验。

### 6.9 账号与云同步

主项目已包含账号同步入口。

目标是支持用户换电脑或手机后恢复：

- 课程表
- 打卡记录
- 错题集
- 明日重点
- 每日提醒设置
- 奖励领取状态

当前原型包含：

- 邮箱登录
- 密码登录
- 登录后自动同步
- 立即同步
- 退出登录
- 未登录时可先体验
- 最近同步时间展示

文案方向：

```text
换电脑或手机登录同一账号，即可恢复课程表、打卡和错题集。
```

### 6.10 数据保存与隐私边界

当前原型主要使用本地存储保存学习状态，并在登录后同步到云端。

需要保存的数据包括：

- 用户设置
- 课程表任务
- 打卡记录
- 错题集
- 明日重点
- 每日提醒设置
- 奖励状态

需要注意：

- 本地上传的资料文件刷新后需要重新选择
- 云同步保存学习状态，不应在未说明的情况下长期保存原始本地文件
- AI 搜索和资料核验需要展示来源，避免用户误以为所有内容都已被官方确认

### 6.11 当前主站奖励原型

主站根目录当前仍保留一套早期奖励原型，包括：

- 首次打卡奖励
- 7 天打卡奖励
- 14 天打卡奖励
- 30 天打卡奖励
- 首次错题解决奖励
- AI 视频形式的泰蛋亮灯素材
- 小卡、歌词卡、票根等奖励探索

这些是早期探索版本，视觉和架构上与新的 `reward-companion-system` 不完全一致。

后续方向：

- 主站继续负责学习流程、打卡和奖励触发判断
- `reward-companion-system` 逐步成为统一奖励表现层
- 旧的 modal 奖励原型应逐步迁移为全屏 RewardScene
- 奖励文案和奖励类型最终应统一由 rewardConfig 管理

## 7. 奖励系统需求

### 7.1 奖励系统定位

奖励系统不是弹窗，不是普通「恭喜完成」提示，而是一个全屏情绪场景。

用户完成学习后，进入紫色演唱会应援舞台，获得像追星一样的应援反馈。

### 7.2 奖励触发场景

当前讨论过的触发节点包括：

- 首次打卡成功
- 连续 7 天打卡
- 连续 14 天打卡
- 连续 30 天打卡
- 连续 50 天打卡
- 连续 100 天打卡
- 连续 365 天打卡
- 错题解决
- 高难题通过
- 今日学习完成
- 明日重点生成

### 7.3 奖励表现方向

可用奖励形式：

- 应援灯
- 小卡
- 双面小卡
- 票根
- 歌单
- 播放器
- 应援旗
- 纪念海报
- 紫色灯海
- 隐藏属性图
- 泰蛋灯光动画

### 7.4 泰妍与泰蛋元素使用原则

奖励系统可以融合：

- 紫色应援色
- 泰蛋
- 应援棒
- 蝴蝶元素
- 隐藏属性图
- 演唱会灯海
- K-pop 应援氛围

但视觉表达应偏二创、氛围化、陪伴化，避免过度商业化或过于直白复刻。

## 8. 奖励系统架构

奖励系统必须保持数据驱动。

### 8.1 统一入口

```text
RewardScene
```

`RewardScene` 是唯一入口。

### 8.2 当前结构

```text
RewardScene
├── Day1Scene
├── Day7Card
├── Day14Player
├── Day30Ticket
├── Day50CheerFlag
├── Day100Poster
└── Day365Ocean
```

### 8.3 架构规则

- 所有奖励内容由 `rewardConfig.ts` 驱动
- 不允许每个 Day 组件自己写死文案
- 不允许多个 rewardConfig
- 不允许在各个 Day 里分散写业务判断
- Day 组件只负责渲染对应奖励形态
- `RewardScene` 负责统一选择奖励场景

### 8.4 当前实现状态

| 奖励节点 | 组件 | 状态 |
|---|---|---|
| Day1 | Day1Scene | 已完整实现 |
| Day7 | Day7Card | 已实现紫色应援歌词卡 |
| Day14 | Day14Player | 占位 |
| Day30 | Day30Ticket | 占位 |
| Day50 | Day50CheerFlag | 占位 |
| Day100 | Day100Poster | 占位 |
| Day365 | Day365Ocean | 占位 |

### 8.5 奖励领取状态

当前阶段已提供最小可用的本地领取状态模块：

```text
reward-companion-system/src/rewards/rewardStorage.ts
```

本地存储 key：

```text
purple-cheer-collected-rewards
```

- 紫色光波扩散
- 星星粒子上升
- RewardScene 淡出

独立 demo 中没有主站页面作为底层页面。

## 13. 与主站的接入方式

主站在用户完成学习行为后，根据奖励规则决定是否打开 RewardScene。

当前子项目已提供本地最小接入方式：

```ts
getNextRewardDay(studyDays)
collectReward(rewardDay)
isRewardCollected(rewardDay)
resetCollectedRewards()
```

其中 `getNextRewardDay(studyDays)` 会根据学习天数和本地领取记录返回下一个应展示的奖励节点；如果没有可领取奖励，返回 `null`。

示例：

```tsx
const rewardDay = getNextRewardDay(studyDays)

if (rewardDay !== null) {
  setActiveRewardDay(rewardDay)
}

<RewardScene
  rewardDay={activeRewardDay}
  onCollect={() => {
    collectReward(activeRewardDay)
  }}
  onClose={() => {
    // 关闭奖励场景
  }}
  onComplete={() => {
    // 返回学习页或继续后续流程
  }}
/>
```

接入原则：

- 主站负责判断是否触发奖励
- RewardScene 负责展示奖励
- rewardConfig 负责提供文案和奖励类型
- Day 组件只负责视觉表现
- `onCollect` 负责记录奖励已领取
- `onClose` / `onComplete` 负责关闭奖励场景或回到学习页

开发环境中提供了一个示例组件：

```text
reward-companion-system/src/examples/StudyCompleteDemo.tsx
```

它用于模拟：

- 当前学习天数
- 点击「完成今天学习」
- 触发奖励判断
- 点击「收下」后写入本地领取记录
- 没有奖励时显示普通完成提示

该测试入口只在 `import.meta.env.DEV` 时显示，生产环境不显示调试入口。

## 16. 验收标准

### 16.1 主流程

- 用户可以完成学习信息填写
- 用户可以生成学习计划
- 用户可以每日打卡
- 系统可以基于打卡生成明日重点
- 首次打卡后可触发 Day1 奖励
- 第 7 天可触发 Day7 SSR 歌词卡奖励

### 16.2 Day1 奖励

- 入场动画节奏符合当前时间线
- 横屏和竖屏底图正确切换
- 按钮 2.7s 左右出现
- 点击按钮后有完整反馈
- 可点击开启声音，领取时有轻量点亮音效
- 独立 demo 不黑屏
- 正式接入时可通过回调关闭 RewardScene

### 16.3 技术验收

- `rewardConfig.ts` 是奖励内容唯一数据源
- `RewardScene` 是奖励系统唯一入口
- Day14 到 Day365 保持占位
- 本地领取状态使用 `purple-cheer-collected-rewards`
- 卡片收藏状态使用 `purple-cheer-collected-cards`
- 已领取奖励不会重复弹出
- localStorage 损坏不会导致页面崩溃
- 无生产调试面板
- mobile 不横向溢出
- Safari 底部栏不遮挡按钮
