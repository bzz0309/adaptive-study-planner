更新时间：2026-07-14

本文档负责主站学习流程，不展开奖励视觉细节和错题算法细节。

相关文档：

- 奖励系统：[REWARD_SYSTEM_SPEC.md](./REWARD_SYSTEM_SPEC.md)
- 错题系统：[WRONG_QUESTION_SYSTEM.md](./WRONG_QUESTION_SYSTEM.md)

## 2. 多考试学习规划

主站需要支持多考试和自定义学习项目。

当前范围：

- TOPIK
- IELTS
- 其他自定义考试或学习项目

不同考试对应不同字段和计划结构：

- TOPIK：区分 TOPIK I / TOPIK II，并支持目标等级
- IELTS：区分 Academic / General Training，并围绕听、说、读、写四项规划
- 其他考试 / 项目：支持用户自定义考试名称、学习项目名称、目标和资料

## 3. 学习信息填写

用户需要填写或选择：

- 考试名称：TOPIK / IELTS / 其他
- 考试类别或学习项目类型
- 目标等级、目标分数或目标描述
- 考试日期、考试届次或学习截止时间
- 当前学习基础或备考状态
- 学习强度：轻量、中等、自定义等
- 每天最少 / 最多学习时长
- 可学习时间段
- 每周学习日
- 第一轮计划用时
- 薄弱内容
- 学习资料或资料链接

计划生成时必须考虑用户真实可用时间，避免生成看起来完整但实际不可执行的计划。

## 4. 关键信息核对

生成计划前，系统需要引导用户核对已填写或选择的所有关键信息。

避免过窄表达：

```text
请核对：TOPIK I/II、目标等级、考试日期或届次是否正确。
```

推荐表达：

```text
请核对你刚才填写和选择的关键信息。
```

或：

```text
生成计划前，先一起确认这些信息有没有填对。
```

## 5. 学习课程表生成

系统基于用户的考试目标、备考时间、薄弱内容和资料来源生成学习课程表。

周计划采用一周七列的真实时间表：左侧显示统一时间刻度，任务卡按开始和结束时间定位，使用户能直接看出先后顺序和空档。手机端保持时间轴可见并横向查看各天，不把七列压缩到无法阅读。完成状态必须显示清晰的勾选标记，不能只留下无含义的圆圈。

课程表需要体现：

- 阶段目标
- 每日任务
- 复习安排
- 错题回看
- 考前冲刺节奏

对不同考试，计划内容应按考试结构自动调整：

- TOPIK I：听力、阅读、词汇语法基础
- TOPIK II：听力、写作、阅读、词汇语法基础
- IELTS：Listening、Reading、Writing、Speaking
- 其他考试 / 项目：根据用户填写内容、上传资料和 AI 搜索结果生成模块化计划

## 6. 学习行为记录

学习记录必须由系统从学习行为中生成，不要求用户手动填写完成状态、正确题数或总题数。

系统自动记录：

- startedAt / completedAt
- 任务是否完成
- 系统练习总题数
- 系统判断的正确题数和正确率
- 错题列表与错因
- 学习用时

学习进度必须区分两种口径：

- 任务进度只按周计划任务的真实完成数量计算；部分完成、站外补录、取消和改期不得冒充计划完成。
- 实际学习用时累计站内已完成、站内部分完成及当周站外补录；站外补录只证明发生过学习，不自动匹配或完成某个计划任务。

“学习进度”页应提供简短的计划与实际对照，至少展示计划任务完成数、计划用时、实际学习用时，以及进行中、取消或未完成重排的真实状态。

用户只在完成学习答题后补充可选反思：

- 今天哪里卡住了
- 哪个知识点不确定
- 是否希望明天继续练同类题
- 一句话备注

禁止把以下内容交给用户手填：

- 是否完成
- 正确题数
- 总题数
- 是否打卡

学习完成逻辑：

```text
开始学习
→ 系统出题 / 执行任务
→ 用户作答或完成步骤
→ 系统统计结果
→ 自动写入学习记录
→ 用户可选补充反思
→ 更新今日状态与 streakDays
→ 必要时触发 Reward 反馈层
```

## 7. 明日重点

系统根据当天学习行为记录生成明日重点。

核心任务流：

```text
每天根据系统记录自动生成明日重点
```

明日重点应综合：

- 当天完成情况
- 未完成任务
- 错题复习队列
- 到期变式练习
- 用户薄弱项

### 7.1 系统出题机制

开始学习后，系统出题必须来自任务上下文和错题上下文，而不是让用户手填结果。

当前机制：

- 计划生成时优先调用 `/api/study-assistant` 生成学习任务
- 如果 AI 计划不可用，则使用本地模板 `generatePlanFromSettings(settings)` 生成任务
- 进入练习时，系统先根据用户填写的考试类型、级别、目标等级、当前任务和薄弱项请求出题
- 真实资料题库存在匹配题目时，优先使用真题 / 教材 / 用户资料原题
- AI 主要负责中文释义、答题解析、错因归纳、生词语法整理和少量同型补充题
- 缺少真实资料题时，才参考官方公开样题、公开真题题型和用户资料生成同型练习
- 不把 AI 生成题伪装成真题，不把来源不明题目作为官方题
- `/api/study-assistant` 已支持 `practice` action；如果配置了 `OPENAI_API_KEY`，会尝试在线生成同型题；如果未配置或调用失败，则使用结构化兜底题
- 在线出题与手写识别使用智增增承载的 OpenAI 兼容服务，TTS 使用 MiniMax：
  - `OPENAI_API_KEY`：服务端 API key，不得写入代码或提交到 GitHub
  - `AI_PROVIDER`：默认 `openai`，控制在线出题与手写识别提供方
  - `TTS_PROVIDER`：生产配置为 `minimax`，只控制无原始音频时的朗读回退
  - `OPENAI_BASE_URL`：智增增兼容接口；同时设置 `OPENAI_ALLOW_CUSTOM_BASE_URL=true` 才启用
  - `OPENAI_MODEL`：可选；必须使用服务商后台提供的精确模型 ID
- AI 生成题必须经过质量审查层：检查题干、4个选项、唯一答案、解析长度、题型来源和当前 UI 适配性
- 未通过质检的题不得进入练习；题量不足时使用结构化兜底题补齐
- 用户可以在开始学习前选择本组题数：5 / 10 / 15 / 20；题数只影响练习量，不允许用户手动填写正确题数或完成状态
- 如果在线出题不可用，才使用本地题库数组作为兜底：
  - `particleQuestions`：TOPIK 助词强化题
  - `placeParticleQuestions`：导入错题后的场所助词诊断题
- 系统根据用户选择自动判断答案、统计正确题数、总题数、正确率和错题解释

目标机制：

```text
任务上下文 / 错题上下文
→ 生成练习题
→ 用户作答
→ 系统判题
→ 自动写入学习记录和错题系统
```

题目来源优先级：

1. 用户填写的考试类型、级别和目标等级，例如 TOPIK II 目标4级
2. 用户导入的真题 / 教材 / 学习包资料
3. 用户导入的错题照片、截图或 PDF
4. 官方公开样题、公开真题题型和考试说明
5. 当前任务的学习目标和完成标准
6. 错题系统中的到期复习队列
7. 考试类型与薄弱项模板
8. 本地兜底题库

真实资料题库结构详见：[REAL_MATERIAL_QUESTION_BANK.md](./REAL_MATERIAL_QUESTION_BANK.md)

## 8. 学习提醒

默认提醒时间：

```text
21:30
```

原因：

- 晚上 22:30 对用户可能偏晚
- 21:30 更适合作为轻量提醒
- 提醒语气应柔和，不应制造负担

## 9. 学习资料与 AI 核验

### 9.1 资料输入方式

当前主项目支持：

- 资料链接输入
- 本地资料上传
- 图片、PDF、Word、PPT、表格、文本等文件类型
- 一次最多上传 10 个学习资料文件
- 文件列表展示与删除

本地资料刷新后需要用户重新选择，不应默认长期保存原文件。

### 9.2 AI 搜索补充

当用户资料不足时，可以开启：

```text
缺少资料时让 AI 搜索补充
```

搜索原则：

- 优先考试官网
- 优先权威资料
- 优先题型、考试范围、评分标准、官方样题
- 搜索结果需要展示来源
- 用户确认后才生成计划

### 9.3 资料确认

生成计划前需要进入资料确认步骤。

确认内容包括：

- 考试 / 项目
- 目标
- 学习范围
- 时间安排
- 资料来源
- AI 搜索结果

文案方向：

```text
请最后核对一次关键信息。确认后，AI 会按这些资料生成课程表。
```

如果 AI 搜索仍在进行，应禁用生成按钮，并提示等待资料核验。

## 10. 账号与云同步

主项目已包含账号同步入口。

目标是支持用户换电脑或手机后恢复：

- 课程表
- 打卡记录
- 错题集
- 明日重点
- 每日提醒设置
- 奖励领取状态

当前原型包含：

- 邮箱登录
- 密码登录
- 登录后自动同步
- 立即同步
- 退出登录
- 未登录时可先体验
- 最近同步时间展示

文案方向：

```text
换电脑或手机登录同一账号，即可恢复课程表、打卡和错题集。
```

## 11. 数据保存与隐私边界

当前原型主要使用本地存储保存学习状态，并在登录后同步到云端。

需要保存的数据包括：

- 用户设置
- 课程表任务
- 打卡记录
- 错题集
- 明日重点
- 每日提醒设置
- 奖励状态

需要注意：

- 本地上传的资料文件刷新后需要重新选择
- 云同步保存学习状态，不应在未说明的情况下长期保存原始本地文件
- AI 搜索和资料核验需要展示来源，避免用户误以为所有内容都已被官方确认

## 12. 最终产品主线与反馈层关系

主站是 Study Planner 的核心入口，负责学习任务流：

```text
plan → execute → complete → record
```

Reward Engine 只作为学习完成后的情绪反馈层，不承担主导航、学习计划或错题复习职责。

最终产品架构冻结为：

```text
学习行为（主）
→ 错题强化（学习能力增强）
→ Reward 反馈（情绪反馈层）
→ Cheer Box（成长回顾层）
```

用户唯一主路径：

```text
HomeDashboard
→ 学习执行
→ 错题系统（可选强化）
→ Reward Engine（反馈）
→ My Cheer Box（回顾）
```

连续学习里程碑仍由 Reward Engine Core 判断：

```ts
const rewardDay = getNextRewardDay(streakDays)
```

如果返回 Day1 / Day7 / Day14 / Day30 / Day50 / Day100，则打开反馈场景；如果没有关键节点，则只展示轻反馈或回到主站学习流。

当前已进入产品基线 / 成长展示的连续节点：

| 节点 | 触发 | v2 成长语义 | RewardScene 表现 |
|---|---|---|---|
| Day1 | `streakDays >= 1` | `start / calm / 被看见` | 第一盏应援灯亮起，用户收下首日成长标记 |
| Day7 | `streakDays >= 7` | `stable / excitement / 被记录` | 卡池待机视频、抽卡视频、SSR 正反面小卡和完成态 |
| Day14 | `streakDays >= 14` | `rhythm / stability / 稳定反馈` | 不新增 RewardScene 流程，仅在 My Cheer Box v2 表达稳定阶段 |
| Day30 | `streakDays >= 30` | `identity_forming / pride / 习惯确认` | 不新增 RewardScene 流程，仅在 My Cheer Box v2 表达习惯形成 |

主站只负责学习行为闭环和调用 Reward Engine Core 获取反馈节点，不直接写 Day 节点展示逻辑。具体视觉、状态机和素材规则由 `reward-companion-system` 负责。

系统重心收口后的结构：

```text
Study Planner → Reward Engine feedback → My Cheer Box review
```

主站继续承担学习入口、任务完成、学习记录、错题复习和明日重点；Reward Engine 只在学习完成后作为情绪反馈出现。My Cheer Box v2 只作为成长回顾查看页，不作为主导航入口，不参与 reward logic。

冻结规则：

- `HomeDashboard` 不包含 reward / growth / video 逻辑
- 错题系统不参与 reward logic
- Reward Engine 不新增 reward 类型、DayX 逻辑、video flow 或 state machine
- My Cheer Box 不参与 reward 计算，不升级为交互核心

答题全对和错题解决不进入全屏 RewardScene，而是进入轻反馈队列：

- `highlight_feedback`：答题全对 / 单日高光
- `breakthrough_feedback`：错题解决 / 突破卡点

同一天多个奖励触发时：

1. 先展示连续打卡全屏奖励
2. 全屏奖励完成后展示错题解决轻反馈
3. 最后展示答题全对轻反馈

奖励系统细节详见：[REWARD_SYSTEM_SPEC.md](./REWARD_SYSTEM_SPEC.md)

错题触发细节详见：[WRONG_QUESTION_SYSTEM.md](./WRONG_QUESTION_SYSTEM.md)

## 13. 当前主站奖励原型

主站根目录仍保留一套早期奖励原型，包括：

- 首次打卡奖励
- 7 天打卡奖励
- 14 天打卡奖励
- 30 天打卡奖励
- 首次错题解决反馈
- AI 视频形式的泰蛋亮灯素材
- 小卡、歌词卡、票根等奖励探索

这些是早期探索版本，视觉和架构上与新的 `reward-companion-system` 不完全一致。

冻结后的方向：

- 主站继续负责学习流程、打卡和反馈触发
- `reward-companion-system` 只作为反馈表现层存在
- 旧的 modal 奖励原型只允许迁移为轻反馈或现有 Day1 / Day7 反馈，不新增 reward flow
- 反馈文案最终应统一由配置管理，禁止新增奖励类型

## 14. 学习行为驱动的动态排期与当前任务聚焦

本节为后续实现规范，未完成能力不得按已上线表述。

### 14.1 计划调整数据来源

计划调整必须基于开始与完成时间、答题结果、正确率、用时、错题、生词、听写或背写、写作提交，以及改期、取消和未完成状态。

### 14.2 已选范围约束

用户只选择听力和阅读时，系统只安排听力、阅读及由真实结果触发的复盘。词汇和语法不得作为独立计划任务出现，除非用户明确添加。

### 14.3 动态周复盘

周日不固定安排“错题 + 生词整理”。系统根据本周真实错题、生词、完成和未完成任务生成复盘或补做建议；没有记录时，只展示下周准备，不得虚构复盘内容。

### 14.4 编辑、取消与状态同步

任务编辑、取消和改期必须同步到课程表、时长统计、学习进度、明日重点和复盘建议。取消任务保留在历史中，但不得计为完成。

### 14.5 当前页面信息密度

默认页面一次只突出一个当前任务，最多展示三段短信息：现在做什么、为什么现在做、预计多久。页面只保留一个主操作，通常为“开始学习”；详情由用户主动展开。

当前任务卡、周计划卡和任务详情是不同层级：当前任务卡负责系统决策与直接开始；周计划卡负责时间定位；任务详情只补充题数、编辑/重排和已经产生的真实记录。连续两层不得再次完整复述模块、标题、目标和自动记录说明。“调整计划”直接进入编辑，普通任务“开始学习”直接进入练习。

系统生成或兼容旧计划标题时，任务标题、训练重点和 TOPIK 题型编号必须按同一分类内的顺序匹配，不能使用整周任务序号分别套用不同模板。

“明日复习重点”只有在真实答题或错题数据能够形成结论时才显示，并紧跟当前任务作为次级提示；无真实结论时不显示空卡片。每日复盘提醒属于学习设置，不长期占用首页内容区。

### 14.6 在线出题等待与降级

在线出题必须显示真实等待秒数和当前阶段，不使用无法判断是否卡住的假进度。提供方未在内部时限完成时，应在部署平台硬超时之前主动切换到系统题库，不进行另一轮同等时长的静默重试。降级后的题目必须保持当前考试、等级和模块一致，并把标题题量修正为实际可用数量。

## 15. 错题诊断与同型补强

### 15.1 错题记录

每条新错题必须保存可复盘的原题快照：题图、题目要求、听力原文或阅读原文、全部选项、用户选择和正确答案；有原始音频时继续保留原始音频路径。诊断默认只给“错在哪里 / 正确思路 / 改进建议”，其中改进建议必须引用当前题的关键词、错误选项与正确选项，不能只给通用学习口号。

错因必须来自当前题型和真实作答证据。词义、生词或同义辨析题可归为“生词影响”；只有单题在页面可见状态下实际停留超过3分钟，才归为“时间分配问题”，切换标签页的时间和听力播放等待不计入。没有足够证据时继续使用题目理解、知识点或粗心等已有归因，不猜测用户状态。

练习提交后的“题目要求”必须解释当前题在问什么，不能用整段听力译文或阅读译文代替。听力原文与中文释义只展示一次，随后再展示围绕当前答案依据的答题解析；阅读题仅在存在独立原文释义时显示原文区块。

系统听力兜底不得只更换题干和选项而重复使用同一段原文。每题的原文、中文释义、朗读文本、题干、选项和答案必须一一对应；只有明确呈现为“同一材料的第 1/2 题”时才允许共享原文。

听写页的第一步必须是清晰的音频播放器，展示音频准备、播放中、播放完成与失败状态。答案提示不得占据主要视觉面积。手机和平板继续支持系统韩文手写键盘；电脑端可在浏览器提供韩文手写模型时把画布识别结果回填答案框，缺少模型时必须明确说明并保留系统韩文输入法路径。

### 15.2 触发规则

首次出现的错题只重做保存的当前原题，不调用在线服务生成额外题。同类错因第 2、3、4 次及以上出现时，才依次提供 5、10、20 道同型补强题，并把结果用于后续计划调整。不得每错一题就机械生成大题包，也不得用随机泛化练习替代针对性补强。

### 15.3 反馈表达

错题反馈先给结论与可执行动作，避免长篇泛泛聊天；更多解释由用户主动查看。

## 16. 听写、背写与写作复盘

### 16.1 听写与背写

听写或背写是独立的学习行为，优先覆盖真实材料中的常见单词和短句。后续支持 iPad 或 iPhone 手写输入、键盘输入兜底、核对反馈和错词归档。

### 16.2 写作记录

写作提交应进入学习记录与诊断，形成具体的错误点、正确表达和下一步练习，而非只作为自由反思文本。

### 16.3 记录用途

听写、背写、写作与错题结果共同驱动后续复盘和动态排期。
