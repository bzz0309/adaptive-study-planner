更新时间：2026-06-25

本文档负责主站学习流程，不展开奖励视觉细节和错题算法细节。

## 2. 多考试学习规划

主站需要支持多考试和自定义学习项目。

当前范围：

- TOPIK
- IELTS
- 其他自定义考试或学习项目

不同考试对应不同字段和计划结构：

- TOPIK：区分 TOPIK I / TOPIK II，并支持目标等级
- IELTS：区分 Academic / General Training，并围绕听、说、读、写四项规划
- 其他考试 / 项目：支持用户自定义考试名称、学习项目名称、目标和资料

## 3. 学习信息填写

用户需要填写或选择：

- 考试名称：TOPIK / IELTS / 其他
- 考试类别或学习项目类型
- 目标等级、目标分数或目标描述
- 考试日期、考试届次或学习截止时间
- 当前学习基础或备考状态
- 学习强度：轻量、中等、自定义等
- 每天最少 / 最多学习时长
- 可学习时间段
- 每周学习日
- 第一轮计划用时
- 薄弱内容
- 学习资料或资料链接

计划生成时必须考虑用户真实可用时间，避免生成看起来完整但实际不可执行的计划。

## 4. 关键信息核对

生成计划前，系统需要引导用户核对已填写或选择的所有关键信息。

避免过窄表达：

```text
请核对：TOPIK I/II、目标等级、考试日期或届次是否正确。
```

推荐表达：

```text
请核对你刚才填写和选择的关键信息。
```

或：

```text
生成计划前，先一起确认这些信息有没有填对。
```

## 5. 学习课程表生成

系统基于用户的考试目标、备考时间、薄弱内容和资料来源生成学习课程表。

课程表需要体现：

- 阶段目标
- 每日任务
- 复习安排
- 错题回看
- 考前冲刺节奏

对不同考试，计划内容应按考试结构自动调整：

- TOPIK I：听力、阅读、词汇语法基础
- TOPIK II：听力、写作、阅读、词汇语法基础
- IELTS：Listening、Reading、Writing、Speaking
- 其他考试 / 项目：根据用户填写内容、上传资料和 AI 搜索结果生成模块化计划

## 6. 每日打卡

用户完成当天学习后可以打卡。

打卡不是为了制造压力，而是用于记录：

- 今天完成了什么
- 哪些内容卡住了
- 哪些错题解决了
- 明天应该优先看什么

## 7. 明日重点

系统根据当天打卡结果生成明日重点。

核心任务流：

```text
每天根据打卡自动生成明日重点
```

明日重点应综合：

- 当天完成情况
- 未完成任务
- 错题复习队列
- 到期变式练习
- 用户薄弱项

## 8. 学习提醒

默认提醒时间：

```text
21:30
```

原因：

- 晚上 22:30 对用户可能偏晚
- 21:30 更适合作为轻量提醒
- 提醒语气应柔和，不应制造负担

## 9. 学习资料与 AI 核验

### 9.1 资料输入方式

当前主项目支持：

- 资料链接输入
- 本地资料上传
- 图片、PDF、Word、PPT、表格、文本等文件类型
- 一次最多上传 10 个学习资料文件
- 文件列表展示与删除

本地资料刷新后需要用户重新选择，不应默认长期保存原文件。

### 9.2 AI 搜索补充

当用户资料不足时，可以开启：

```text
缺少资料时让 AI 搜索补充
```

搜索原则：

- 优先考试官网
- 优先权威资料
- 优先题型、考试范围、评分标准、官方样题
- 搜索结果需要展示来源
- 用户确认后才生成计划

### 9.3 资料确认

生成计划前需要进入资料确认步骤。

确认内容包括：

- 考试 / 项目
- 目标
- 学习范围
- 时间安排
- 资料来源
- AI 搜索结果

文案方向：

```text
请最后核对一次关键信息。确认后，AI 会按这些资料生成课程表。
```

如果 AI 搜索仍在进行，应禁用生成按钮，并提示等待资料核验。

## 10. 账号与云同步

主项目已包含账号同步入口。

目标是支持用户换电脑或手机后恢复：

- 课程表
- 打卡记录
- 错题集
- 明日重点
- 每日提醒设置
- 奖励领取状态

当前原型包含：

- 邮箱登录
- 密码登录
- 登录后自动同步
- 立即同步
- 退出登录
- 未登录时可先体验
- 最近同步时间展示

文案方向：

```text
换电脑或手机登录同一账号，即可恢复课程表、打卡和错题集。
```

## 11. 数据保存与隐私边界

当前原型主要使用本地存储保存学习状态，并在登录后同步到云端。

需要保存的数据包括：

- 用户设置
- 课程表任务
- 打卡记录
- 错题集
- 明日重点
- 每日提醒设置
- 奖励状态

需要注意：

- 本地上传的资料文件刷新后需要重新选择
- 云同步保存学习状态，不应在未说明的情况下长期保存原始本地文件
- AI 搜索和资料核验需要展示来源，避免用户误以为所有内容都已被官方确认

## 12. 主站如何触发奖励系统

主站在用户完成学习行为后，判断是否触发奖励。

连续学习里程碑使用全屏 RewardScene：

```ts
const rewardDay = getNextRewardDay(streakDays)
```

如果返回 Day1 / Day7 / Day14 / Day30 / Day50 / Day100，则打开 RewardScene。

答题全对和错题解决不进入全屏 RewardScene，而是进入轻反馈队列：

- `highlight_feedback`：答题全对 / 单日高光
- `breakthrough_feedback`：错题解决 / 突破卡点

同一天多个奖励触发时：

1. 先展示连续打卡全屏奖励
2. 全屏奖励完成后展示错题解决轻反馈
3. 最后展示答题全对轻反馈

奖励系统细节详见：[REWARD_SYSTEM_SPEC.md](./REWARD_SYSTEM_SPEC.md)

错题触发细节详见：[WRONG_QUESTION_SYSTEM.md](./WRONG_QUESTION_SYSTEM.md)

## 13. 当前主站奖励原型

主站根目录仍保留一套早期奖励原型，包括：

- 首次打卡奖励
- 7 天打卡奖励
- 14 天打卡奖励
- 30 天打卡奖励
- 首次错题解决奖励
- AI 视频形式的泰蛋亮灯素材
- 小卡、歌词卡、票根等奖励探索

这些是早期探索版本，视觉和架构上与新的 `reward-companion-system` 不完全一致。

后续方向：

- 主站继续负责学习流程、打卡和奖励触发判断
- `reward-companion-system` 逐步成为统一奖励表现层
- 旧的 modal 奖励原型应逐步迁移为全屏 RewardScene 或轻反馈
- 奖励文案和奖励类型最终应统一由配置管理
