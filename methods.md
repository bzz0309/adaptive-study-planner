# 工作方法

## 编辑计划的完成路径

- 单项计划保存成功后结束编辑上下文，不重新打开任务详情。
- 回到周计划并定位到修改后的任务卡片，让用户立即确认日期、时间和内容已经变化。
- 定位同时兼顾键盘焦点与减少动态效果偏好，避免只依靠短暂动画传达结果。

## 1. 修改前：先确认真实数据路径

1. 查看工作区状态，识别用户现有改动。
2. 从页面状态追到数据来源，区分真实答题记录、旧 localStorage、示例数据和 AI 回退数据。
3. 只修改当前问题所在的路径，不借机调整无关模块。

## 2. 练习题验收

题目进入页面前逐项检查：

- 来源类型与标签一致：真题/教材、AI 补充或本地回退。
- 图、音频、题干、选项、正确答案互相匹配。
- 提交后按“正确选项 -> 题目要求 -> 原文与中文释义 -> 答题解析”展示；题目要求必须解释问题本身，原文及其中文释义不得重复出现。
- 阅读题提交前检查“题目要求 -> 阅读原文 -> 选项”是三个清晰层级；用旧式 `stem` 同时包含要求和原文的数据回归，确认页面会自动拆分，错题快照也分别保存。
- 错题只在真实答错后入库，复习建议引用真实错题或下一次任务，不使用预置成绩。
- 新增错题时检查持久化数据中包含完整题目快照、用户选项与正确选项；刷新后错题页仍能还原原题。
- 检查“改进建议”是否点名当前题的关键词或选项差异，并给出下一次可执行动作；“多听、多看、注意关键词”不能单独作为建议。
- 完成错题复习后分别验证未通过和通过：卡片当天显示已复习，待复习数量减少；未通过保留当前阶段并安排次日重做，通过推进到下一个 1/3/7/14 天节点；刷新后状态仍存在。
- 在线出题需分别测试正常返回、提供方超时和平台中断：超时必须在平台硬时限之前主动切换题库，页面显示真实等待秒数，最终题量和分类以实际可用结果为准。至少用“请求 10 题、兜底只有 5 题”回归一次：接口须返回 200，首页目标、弹窗总数、暂停后的剩余数都应统一为 5。

## 3. 听力验收

1. 有 `audioSrc` 时必须先播原始音频。
2. 无原始音频时才请求 TTS；对话按角色拆分，不能用单一旁白替代。
3. 验证按钮的生成中、播放中、失败提示及重试。
4. 用真实对话题与非对话题各测一次，确认可听、题目内容匹配、状态可恢复。
5. 连续检查整组题目的 `transcript`/`audioText`；除显式的同材料题组外，每题原文应不同，不能只替换题干和选项。
6. 听写音频分别检查准备中、播放中、播放完成和失败重试文案；播放按钮在状态变化后仍可重听。
7. 手写识别分别测试浏览器支持韩文模型和不支持两条路径：支持时识别结果回填答案框并可切换候选，不支持时给出明确输入法提示。
8. 在 iPad/iPhone 路径检查答案卡的标题、绿色留白和中央输入面均可聚焦同一个原生韩文输入框；系统手写写入后应直接参与核对。点击“手写输入”展开的是电脑端识别区，两条路径不得互相覆盖。
9. 听写核对必须分别测试正确和错误：结论、卡片颜色、正确答案和复习清单状态一致，不能用“再看看”等含糊文案代替判断。
10. 有不熟词时，从右上角进度卡进入重做，确认跳到真实 `weakIds` 题目并清空旧答案；没有不熟词时入口必须禁用。
11. 单词消除分别从韩文和中文开始配对，验证正确时只清除同 ID 两张卡、错误时不清除、进度只在正确配对后增加；重绘时未消除卡片顺序保持稳定。
12. 从周计划进入听写时，验证任务 ID、组内词条和开始时间已保存；中途离开后状态为部分完成、实际用时增加，再次进入从原位置继续。
13. 完成最后一个词后验证正确率、不熟词、完成时间和实际用时写回任务；首页切换到下一项，周计划显示完成，明日重点使用“个词”而非“道错题”。单独点击顶部听写入口时不得生成计划打卡。
14. 桌面宽屏检查听写工作区水平居中，标题、进度卡与练习卡处于同一最大宽度；右侧不得出现明显失衡空区。820px 与 390px 下恢复满宽单列且无横向溢出。

## 4. 计划验收

- 计划只从用户选择的专项、可用时间、目标等级和实际学习结果生成。
- 计划设置只放影响计划生成的字段；提醒、补录、同步等日常工具必须从独立入口进入。
- 设置流程按“学习目标 -> 学习时间 -> 补充资料”逐步验证。资料为空时可以直接生成；返回上一步不能清空已填写内容。
- 点击生成后分别验证较慢响应、在线回退和真正异常：进度阶段始终在视口内；成功返回周计划；失败恢复“生成周计划”按钮并保留原学习记录。
- 用“每天任务数恰好等于所选模块数”的设置回归排课：同一模块必须在不同日期轮换学习时段；已有真实弱项时，其周频次应高于其他模块，但已完成、部分完成、取消和调整记录不重新排时。
- 用 11:00–21:00 高强度设置检查稳定错峰：一周应出现多个不同开始时间，16:00–18:00 可用时段应被利用；任何任务不得跨越 12:00–13:30、18:00–19:00，任务之间至少保留 10 分钟。
- 修改整体设置后，完成、部分完成、取消和调整轨迹继续保留；新计划只替换没有真实学习轨迹的任务，保存后回到周计划。
- 不把未选择的专项加入计划。
- 周末复盘、错题和生词内容依据真实打卡与错题，不固定预填。
- 编辑、取消和完成状态要同步反映在周计划、进度和复盘入口。
- 动态调整至少覆盖三条真实分支：部分完成后继续置顶且完成数不增加；低于 80% 时将现有同类剩余任务标为系统优先并显示原因；跳过后移动到下一个学习日并保留原日期与调整记录。
- 达到目标正确率时不应机械加题；没有同类剩余任务时只把真实错题交给复习系统，并明确周计划没有被虚构扩充。
- 时间表需验证任务卡顶部位置对应开始时间、高度对应实际时长；连续任务之间不重叠，左侧时间刻度在桌面和手机横向滚动时都可辨认。
- 分别用已完成和未完成的跨模块任务验证模块筛选：筛选后只能存在目标模块卡片；再次点击当前筛选后全部恢复。状态只能改变卡片表现，不能让非目标模块继续显示。
- 从首页“开始学习”验证普通任务直接进入练习；从“调整计划”验证直接进入编辑；从周计划打开任务时，详情不重复模块、标题和空的自动记录说明。
- 用至少两个同分类任务检查标题、训练说明和对应题号，显示顺序必须按分类内索引一致，不能受其他分类任务插入位置影响。
- 补录至少测试“25分钟词汇”“半小时单词和20分钟听力”“听力20分钟，单词半小时”“未写用时”和“无法识别模块”五类输入：多项学习必须先按逗号等语句边界匹配各自模块，再分别显示与合计；只有可识别的真实模块与用时才能改变当前推荐，且推荐变化必须保留原计划任务和完成数。
- 补录后检查首页持续显示“实际学习增加多少”和“当前推荐变为哪项或为何不变”；点击“查看记录”应进入学习进度并看到相同原话、用时和规划影响。
- 学习进度的实际用时必须能核对为“站内计时 + 本周站外补录”；站内来自任务开始至完成或暂停的累计时长，补录来自文本中的小时/分钟，不得把计划时长计入实际。
- 语音补录测试点击麦克风开始、再次点击停止、浏览器拒绝权限和不支持网页语音四条路径；识别文字仍允许修改后再确定。

## 5. 交付前

- 检查差异和格式问题。
- 运行与改动范围相称的检查。
- 用浏览器实际走一遍主要用户路径，而不只看静态页面。
- 在 `worklog.md` 和 `handoff.md` 记录真实完成状态和未验证项。

## 6. 生产站真实点击验收

1. 内置浏览器不可用时，使用项目内的微软官方 `playwright-cli` Skill，不用静态代码检查代替交互验收。
2. 每次创建新的隔离会话，不连接个人浏览器，不读取用户 Cookie、账号或已有学习数据。
3. 至少覆盖桌面与 390px 手机尺寸，检查页面可打开、主操作可点击、无横向溢出，并保存必要的临时截图人工查看。
4. 涉及补录和计划时，用隔离测试数据走完“生成计划 -> 补录 -> 查看记录 -> 返回周计划”，确认补录只增加实际用时，不增加任务完成进度。
5. 检查浏览器控制台与失败提示；`.playwright-cli/` 快照仅用于本地验收并保持忽略。
6. 错题路径必须分别测试：首次错题点击后只打开原题且不请求在线出题；同类错因第 2、3、4 次分别显示 5、10、20 题；复习未通过、通过、14天掌握三种状态刷新后仍保留。

## 7. 视觉规范验收

1. 先检查信息层级，再检查装饰：当前任务、完成标准和主按钮必须先于辅助说明被看到。
2. 核对 `visual-system.css` 中的字体、间距、颜色、圆角和按钮状态；新增页面不得在业务样式里另起一套视觉变量。
3. 桌面检查 1440px，平板检查 820px，手机检查 390px；三档都要覆盖首页、学习进度、任务弹窗和补录弹窗。
4. 悬停、键盘焦点、按下、禁用和加载状态逐一检查，尤其防止白底白字、空白圆圈、看似可点但无响应。
5. 课程表颜色用于模块识别，可以保留多色；页面结构、按钮和弹窗不得继续扩展额外强调色。
6. 浏览器验收受运行环境阻断时，记录具体阻断点并暂停生产部署；构建、静态资源和语法检查只能记为工程检查通过。

## 8. 韩文手写识别回退

1. 先尝试浏览器 `Handwriting Recognition API`，本地模型有结果时不上传画布。
2. 本地 API 缺失、韩文模型不可用或未返回候选时，才调用 `/api/study-assistant` 的 `handwriting` 动作。
3. 上传前把透明画布压成白底 JPEG 并限制宽度；服务端限制 Data URL 类型和体积，只返回 1 至 3 个韩文候选，不保存图片。
4. 在线识别按钮必须持续显示处理中状态；成功回填真实答案框，失败允许清空重写或继续使用系统韩文输入法。

## 9. 智增增 OpenAI 与 MiniMax 提供方回归

1. 同时模拟存在 OpenAI、Qwen、MiniMax 和 ElevenLabs 配置；`AI_PROVIDER=openai` 且开启自定义地址时，在线出题和手写识别必须只请求 `OPENAI_BASE_URL` 的 OpenAI 兼容接口，不能请求 Qwen。
2. `TTS_PROVIDER=minimax` 时，语音必须只请求 MiniMax；对话分别传入 male、female、male2、female2，检查角色 voice 映射和响应头 `X-TTS-Provider=minimax`。
3. 真实题目有 `audioSrc` 时不调用 TTS；只有缺少原始音频才进入 MiniMax 朗读回退。
4. 正式部署后分别请求出题、手写和 TTS 端点；只记录提供方名称、状态码和安全错误类型，不输出密钥、兼容地址或完整鉴权响应。

## 10. 听写状态恢复回归

1. 用全新听写状态进入页面，结果卡不得出现，主操作应为“核对”。
2. 注入旧异常状态 `revealed=true` 且答案为空后重新进入，仍应按未作答显示，不能出现“答案不正确 / 未输入”。
3. 输入正确或错误答案并核对后刷新，重新进入应恢复当前词的答案与核对结果。
4. 点击上一条、下一条或不熟词重做时，未作答的新词不得继承上一词的结果；周计划听写只恢复当前词在任务会话中已保存的答案。

## 11. 在线出题等待回归

1. 模拟提供方持续不返回，5 题请求必须在 12 秒服务端上限后返回系统题库，状态为 HTTP 200，不让前端等待到平台硬超时。
2. 前端请求只比服务端多保留少量网络缓冲；加载文案必须明确最长等待时间，超过上限显示正在切换题库。
3. 超时兜底仍需核对考试、等级、模块和实际题数；不能为了速度混入其他模块或返回空题组。

## 12. 周计划完成卡视觉回归

1. 同时准备一张有真实完成记录的任务和一张待完成任务；完成卡必须明显变淡，待完成卡保持完整模块色。
2. 完成卡仍需显示完成勾、保留可读标题并可点击查看记录；悬停和键盘聚焦后仍低于待完成卡的视觉权重。
3. 桌面和 390px 手机分别检查时间轴定位与横向溢出，淡化状态不得改变卡片尺寸或位置。

## 13. 错因证据与单题计时回归

1. 普通阅读错题在正常作答时间内应归为题目理解错误；词义、生词、近反义或韩文词汇题应归为生词影响。
2. 非听力题在页面可见状态下单题停留超过3分钟，应归为时间分配问题，并给出60秒定位、90秒标记后跳过的具体动作。
3. 切换到其他标签页期间停止累计单题时间；听力题即使播放或等待超过3分钟，仍优先依据听力内容诊断，不误判为时间问题。
4. 完成题组后检查错题卡保存原题快照、实际用时、错因、当前选项差异与改进建议；刷新后不得丢失。

## 14. 历史错题照片导入回归

1. 使用真实图片请求 `vision` 动作，确认服务端只调用当前 OpenAI 兼容视觉模型；无配置或超时时明确失败，不回退到固定假结果。
2. 分别测试清晰题、答案不可见题和多题照片；不可见字段保持空值并标记需要确认，不允许补猜。
3. 导入前修改题干、用户答案、正确答案和错因，确认错题卡显示修改后的原题、全部选项与答案标记。
4. 点击首次复习应直接打开保存的1道原题，且不请求在线出题；刷新后题目快照、到期状态和来源仍存在。
5. 在390px检查勾选框、表单双列转单列、底部确认按钮和弹窗滚动；不得出现全局输入样式把复选框放大或文字竖排。

## 15. 全站视觉系统回归

1. 先在生产或本机真实页面截图审计，再改 `visual-system.css`；不根据单张设计稿猜测动态数据页面。
2. 设计参数默认使用变化度5、动效3、密度4：一套结构主色、一种表面圆角、一种控件圆角，模块色只表达课程类别。
3. 1440px检查内容宽度、工具组、当前任务和日历基线；820px检查顶部收缩、分步弹窗与粘性操作；390px检查换行、主按钮、弹窗底部和页面级横向溢出。
4. 对主按钮实际悬停并读取计算颜色，避免白底白字；对系统深色模式实际模拟截图，不只声明变量。
5. 完成态需要同时降低透明度和饱和度，但不得隐藏文字、完成勾或详情入口；键盘聚焦时仍不能恢复成待完成层级。
6. 静态本机预览的 `/api/cloud-config` 404可单独记录；生产必须再确认函数路由和错误级控制台为0。

## 16. 多周计划数据与回归

1. 不用“当前显示周”推断任务属于哪周；每项任务持久化 `weekIndex`，旧数据缺失时只安全归入第1周。
2. 生成计划时按 `firstRoundWeeks` 建立完整周序列；第1周跳过今天之前且没有真实轨迹的日期，后续周只使用用户勾选的学习日。
3. 所有按 `day` 过滤或判断先后顺序的逻辑都同时使用 `weekIndex` 或完整计划日期，重点检查筛选、统计、当前任务、结果优先级和跳过重排。
4. 重排避开同周同日同时间的现有任务，并记录 `fromWeekIndex/fromDay/toWeekIndex/toDay`；最后一周没有可用位置时保留原任务，不产生范围外的幽灵周。
5. 隔离浏览器至少验证：第1/中间/最后一周标题与任务不同、末页按钮禁用、分类筛选不混入其他类别、旧完成记录升级不丢失、周末手机首屏自动看到今天。
6. 持久化计划周一的 `planStartDate`，并模拟跨自然周重新打开；当前显示周应前进，但每周日期不得随刷新整体漂移。

## 17. 时间线卡片状态矩阵回归

1. 不只抽查一张待完成卡；至少交叉计划中、已完成、部分完成、已取消、学习中与40、45、60分钟三档时长。
2. 计划中卡片固定为“时间和模块时长 / 标题 / 具体计划注释”；其他状态固定为“时间和模块时长 / 标题 / 真实状态”，不得因卡片更高而混入另一种信息。
3. 逐个读取可见子元素边界，确认不超过卡片的上下左右；同时检查顶栏两个字段不折行、不产生内部横向滚动。
4. 长模块名在日历内使用不歧义的短标签，标题不重复顶栏已经表达的模块名；完整名称仍保留在任务详情中。
5. 完成态需要变淡但保持可读，取消状态只出现一次；完成勾和状态文案不得互相覆盖。
6. 再验证分类筛选只生成选中类别的卡片，完成、部分、取消等状态不得绕过类别筛选。
7. 桌面与390px分别复验；手机允许日历容器内部横向滚动，但页面本身不得横向溢出，卡片必须留在所属日期列内。

## 18. 时间线卡片悬停说明回归

1. 卡片布局、字号、圆角、完成态和悬停态必须只有一个最终样式来源；发现 `styles.css` 与视觉覆盖层重复选择器时先合并，避免加载顺序改变结果。
2. 卡片高度继续按真实分钟比例计算，不用固定高度伪装一致；内部只显示能安全容纳的一行注释。
3. 完整说明使用挂在 `body` 下的浮层，避免被卡片的 `overflow:hidden` 裁切；内容使用 `textContent` 写入，不拼接用户数据 HTML。
4. 同时验证鼠标进入/离开、键盘聚焦/失焦、历史数字型与文本型任务 ID、完成态隐藏注释以及长文本边界。
5. 在1440px和390px读取浮层边界，确认不超出视口；手机页面本身不得横向溢出，日历内部横向滚动仍保留。

## 19. 全学习闭环一键回归

1. 先保存回退提交和标签，再用干净的 `localStorage` 启动测试；全新用户必须是零任务、零进度，不能依赖演示数据通过页面渲染。
2. 使用 `specs/full-regression.cli.js` 在同一隔离浏览器会话依次验证练习部分完成与恢复、整组完成、错题快照、筛选、听写、单词消除、补录和提醒，避免多个短会话丢失状态。
3. 练习至少覆盖一次中途关闭和一次完整提交；中途关闭必须保留剩余原题且不计完成，完整提交才写入正确率和实际用时。
4. 听写至少覆盖首次进入无旧结果、错误结果写回、下一词清空旧面板；单词消除同时覆盖错误不消失和正确配对持久化。
5. 站外补录使用一条含两个模块和两个时长的句子，核对总秒数、分项活动和计划完成数不变；只看到 Toast 不算通过。
6. 本地完整点击通过后再做生产静态版本、关键 API 和移动端冒烟。外部云服务不可达时记录为已知阻塞，只验证错误可理解与本地优先，不使用真实账号或密码尝试绕过。

## 20. Impeccable 视觉强化回归

1. 视觉升级前先创建可回退标签，并保存1440px、820px和390px基线截图；不得在没有对照图时批量改全站样式。
2. 先锁定现有产品色板、字体、圆角和信息架构，再通过比例、字重、间距、留白和状态对比强化层级；禁止把渐变、玻璃、装饰动效或新增配色当成“高级感”。
3. 首页只把当前任务作为视觉焦点，但仍保持浅色工作区；日历、听写、错题和进度页继续服从同一设计系统，不重新发明组件。
4. 时间线任务卡只由 `styles.css` 的组件区控制；视觉覆盖层不得新增 `.task-card` 或 `.timeline-task` 规则，避免再次出现高度、裁切和注释丢失。
5. 每轮视觉修改后运行全学习闭环回归，并实看1440px、820px和390px的首页、日历、听写、错题与弹窗；同时检查悬停、键盘焦点、横向溢出和唯一主操作。
6. 本地静态预览允许 `/api/cloud-config` 404，但生产站必须复核资源版本、关键 API、错误级控制台为0及390px页面级横向溢出为0。

## 21. Day1 / Day7 奖励里程碑回归

1. 奖励触发必须使用北京时间自然日；同一天完成多个任务只记一次打卡，不重复解锁。
2. Day1 使用第一次真实完成记录触发。Day7 必须是以本次完成日期结尾、向前连续 7 个自然日都有真实完成记录，累计但断开的 7 天不得触发。
3. 用三组隔离数据分别验证：首次完成触发 Day1；6 个断续历史日期加今天不触发 Day7；连续 6 天加今天触发 Day7。
4. 主站只决定解锁节点和打开哪个奖励页；Day1、Day7 的抽卡、翻面、收下与收藏状态继续由 `reward-companion-system` 管理，不在主站复制状态机。
5. 对 Day1 与 Day7 都实际检查 iframe 参数、关闭路径和重复进入；Day7 还需点击“抽一张 / 查看背面 / 查看正面 / 收下”全流程。
6. 桌面与 390px 手机检查奖励弹窗无页面级横向溢出；正式部署后控制台错误级消息必须为0。

## 22. 扫描真题、题图与逐题音频导入

1. 先用 PDF 渲染逐页定位题号，再分别对照题目卷、听力文本卷和答案卷；扫描件没有可提取文本时不得只依赖 OCR 猜答案。
2. 听力题按同一题号绑定原题裁图、逐题原始音频、韩文原文、中文释义、四个选项和答案；任一项无法确认时不进入启用题库。
3. 题图只裁题目要求和选项，不带页码、下一题残片、中文答案或解析；导入后逐张打开抽查边界。
4. 真实题组必须声明 `matchTerms`，只进入对应训练点；原始 `audioSrc` 永远优先，MiniMax 仅在没有原音频时回退。
5. 阅读题把题目要求、原文和选项分开保存；不得把整段原文拼进大标题。写作长答案在专用输入与评改流程完成前，只登记来源，不伪装成四选一。
6. 每批使用来源清单记录已启用范围和待实现范围，校验稳定 ID、资源存在、原音频唯一性、路由词与答案索引，再跑完整学习闭环。
7. 同一考试模块也必须按原卷的题目要求拆组。例如“对话续句”“听内容一致”“中心思想”不能共用宽泛路由；每组 `matchTerms` 应只命中能真实完成该训练目标的题。
8. 裁图后不能只检查文件存在和尺寸。逐张打开核对题号、四个选项、上边界和下边界；出现上一题选项、下一题标题、页码碎片或任一选项缺失时，必须回到高分辨率源页重新裁切。
9. 同一段官方音频对应多道配套题时，保留各题自己的题图、题干和答案，但按训练点拆到不同任务集合，避免用户在单次练习中误以为内容被重复生成。
10. 原卷出现新的明确题型时，应新增同名训练模板再路由，例如“说话人意图”“职业与身份”；不得为了沿用旧模板把题意改写成“下一步行动”或“人物地点”。
11. 对话中的“中心思想”和“态度”即使共用同一音频，也要按各自题干独立保存答案与解析；态度题必须说明说话人如何回应对方，不能只复述观点内容。
12. “对话前文推断”必须使用独立训练模板，解析要从当前发言中的指代、承接和前提反推上一句；不得与“内容一致”混为同一题型。讲座类中心思想题也要保存当前讲座原文和自己的题图、答案，不能沿用相邻配套题的解析。
13. “说话方式”题使用独立训练模板，解析必须指出说话人采用了概括、分类、原理说明或举例中的哪一种，并引用当前原文的结构证据；不得把它混入“内容一致”或只复述正确选项。
14. 同一段官方长材料的最后两道配套题也要按能力点拆分：内容核对题与说明方式题各自进入对应集合，同时保留相同原始音频路径，禁止为了去重而改成TTS或伪造另一段材料。

## 23. 原生云登录与同步验收

1. 云账号和学习状态使用两个私有路径空间保存；用户文件只保存随机盐密码散列，不保存明文，状态文件由登录令牌中的用户ID定位，客户端不得提交或选择其他用户ID。
2. 原生云提供方必须同时存在私有 Blob 令牌和独立会话签名密钥才启用；密钥只通过 Vercel 环境变量注入，`.env.local`、令牌和测试密码不得进入日志、提交或文档。
3. 注册、登录、刷新、读取用户、退出、保存和恢复都继续适配现有前端接口，避免为修云服务重写学习状态结构或改动既定页面。
4. 私有对象刚写入后可能短暂不可见；登录查用户和跨设备读取状态采用有限退避重试，不无限等待，也不把失败伪装成空状态。
5. 专项验收必须覆盖注册后立即登录、写入可识别的真实学习记录、退出后重新登录恢复、错误密码中文提示和390px弹窗；之后按账号和用户ID计算的精确路径清理测试数据，禁止按整个用户前缀批量删除。
6. 云专项通过后仍需运行完整学习闭环与正式构建；生产部署后再使用新的虚构账号重复同一链路，并确认主域名控制台没有意外错误。

## 24. 单词消除连续分组回归

1. 词库按20词稳定分组，原有第一组不得因扩充而换序；新组只使用已核对的真实教材或真题词汇，并保留来源字段。
2. 错误配对不增加进度，并把两张相关词卡记入本轮错配；正确配对只消除同 ID 的韩文和中文。
3. 完成一组后先读取听写 `weakIds` 和本轮错配，存在真实弱项时先进入不熟词复习；复习结束后才进入尚未完成的新组。
4. 完成页只保留一个主操作，并明确显示组数、当前组进度和总进度；不能自动跳转，也不能把重练旧组写成“新词”。
5. 全部可用词完成后明确写出当前词库总量；重新巩固可以清空当前轮进度，但必须用 `roundsCompleted` 保留完成轮次。
6. 专项测试至少覆盖全部真实分组、真题与教材来源、不熟词优先、新组进入、总量完成、轮次保留及390px两列无页面级横向溢出。

## 25. TOPIK 等级词池回归

1. 听写和单词消除只维护一个当前等级词池入口，两处返回的 ID、顺序和总量必须完全一致，避免两个模块各自扩词后漂移。
2. TOPIK I 词池只接受标注为 TOPIK I 的初级教材词；TOPIK II 词池只接受已核验的 TOPIK II 真题词和中高级教材词，不因数量不足向下混入初级词。
3. 教材 PDF 先渲染并目视核对词条、中文、搭配和例句；同形异义词在配对游戏中只保留一个义项，避免两张韩文卡无法一一对应。
4. 切换等级后重新读取听写与消除状态，确认旧等级的题序、不熟词和已消除 ID 不会串入新等级；切回时仍按对应等级重新建立合法状态。
5. 专项脚本检查两级各自的来源、数量、20词分组、ID/韩文/中文唯一性、共享词池、完成总量和390px布局，再运行完整学习闭环回归。

## 26. 已核验词库自动解锁回归

1. 扩词前先逐页查看教材扫描图，只录入词条、词性与可核对含义；例句使用短学习例句，不批量猜测扫描页内容。
2. 词池按稳定顺序每20词切组，不给分组写死上限。追加已核验词后，总组数和总进度必须由词池长度自动更新。
3. 专项测试直接模拟完成第3组，确认按钮显示“开始下一组”，点击后状态进入 `batchIndex: 3`、页面显示“第 4 组”，且20张新词均来自当前等级的已核验来源。
4. 末组完成时核对总词数、总进度和“从第一组重新巩固”；不能提前宣称全部完成，也不能在词池耗尽后伪造新词。
5. 听写与单词消除继续共用同一词池；扩词后同时更新听写总数并跑完整闭环，防止只在一个模块生效。

## 27. 单词消除音效回归

1. 使用本地 Web Audio 生成短反馈，不新增外部音频文件、网络请求或新的第三方依赖。
2. 错配、正确配对和整组完成必须发出三种可辨别但克制的反馈；整组完成可以在正确音之后追加完成音，不得阻塞卡片消除与状态保存。
3. 页面提供明确的“音效 开/关”按钮并持久化选择；关闭后再次配对不得触发任何音效，重新渲染后仍保持关闭。
4. Web Audio 不可用、系统限制自动播放或用户启用减少动态效果时必须安静降级，视觉反馈和学习记录继续正常工作。
5. 专项回归同时覆盖默认开启、关闭静音、错误/正确区分、整组完成和390px双列布局；之后再运行完整学习闭环回归。

## 28. 生产手写识别与 TTS 回归

1. `package.json` 为 ESM 时先检查所有 Vercel API 路由使用 `export default`；构建成功后仍要直接请求生产接口，避免函数加载失败被前端通用提示掩盖。
2. 手写画布只发送笔迹外接框加少量留白，不能把整块宽画布缩成一张小图；回归测试同时检查裁切后的宽高和页面真实笔迹提交路径。
3. 识别结果至少包含一个预组合韩文音节；只有兼容字母或拆散字母时继续尝试下一个已配置的视觉模型，全部失败才提示重写或改用系统韩文输入法。
4. 使用用户失败样本直接请求生产 `/api/study-assistant`，核对 HTTP 200 和预期完整音节；图片只用于当次请求，不写入仓库或学习记录。
5. 同轮直接请求生产 `/api/tts`，只记录状态码、`Content-Type`、`X-TTS-Provider` 和字节数；必须确认 MiniMax 返回非空真实音频，不输出任何密钥或环境变量值。
